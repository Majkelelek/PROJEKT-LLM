                  PROJEKT AI - RAPORT TECHNICZNY PROJEKTU
=============================================================================

Niniejszy raport zawiera wyczerpujący opis technologii, architektury, mechanizmów
komunikacyjnych oraz komend wykorzystywanych w projekcie Projekt AI.

-----------------------------------------------------------------------------
1. WYKORZYSTANE TECHNOLOGIE I ICH ROLA W PROJEKCIE
-----------------------------------------------------------------------------

A. FRONTEND: React (Vite + Vanilla CSS + JavaScript)
   - Odpowiada za warstwę prezentacji, interakcję z użytkownikiem oraz zbieranie
     danych telemetrycznych z backendu w czasie rzeczywistym.
   - Aplikacja została zorganizowana w modularne komponenty:
     * App.jsx: Główny komponent zarządzający stanem nawigacji, cyklem pętli
       sprawdzającej połączenie (ping) oraz centralnym zarządzaniem historią.
     * Sidebar.jsx: Boczny pasek nawigacyjny z logo, przyciskami zakładek
       (Specyfikacja, Uruchom testy, Raport analityka AI, Zapisane wyniki) oraz
       stopką wyświetlającą status połączenia Ollama i liczbę dostępnych modeli.
     * LoadingView.jsx: Pełnoekranowy ekran ładowania wyświetlany w czasie
       inicjalizacji aplikacji (animowany spinner + komunikat tekstowy).
     * OfflineView.jsx: Ekran diagnostyczny widoczny gdy backend jest niedostępny.
       Wyświetla przyczynę błędu, kod startowy (dotnet run) i przycisk ponowienia.
     * Dashboard.jsx: Kokpit telemetrii. Wizualizuje obciążenie CPU na poziomie
       poszczególnych wątków za pomocą reaktywnych pasków postępu SVG.
     * BenchmarkPanel.jsx: Pozwala wybrać model LLM oraz stopień trudności promptu,
       obsługuje nasłuchiwanie strumienia logów SSE z serwera i wyświetla wyniki
       w postaci pełnej tabeli metryk (czasowe Ollama /verbose, GPU nvidia-smi, RAM/CPU).
     * CompatibilityPanel.jsx: Interaktywny kalkulator szacujący rozkład warstw
       modelu w pamięci VRAM GPU oraz pamięci systemowej RAM (offloading).
     * AIAnalyst.jsx: Prezentuje ustrukturyzowany raport wygenerowany przez LLM
       za pomocą spersonalizowanego parsera składni Markdown.
     * HistoryPanel.jsx: Lista archiwalnych testów pobierana z localStorage.
   - Zarządzanie Historią (localStorage):
     * Cała historia testów jest zapisywana w przeglądarce pod kluczem
       "projekt_ai_history". Dane są zapisywane jako zdeserializowana tablica
       obiektów JSON. Rozwiązanie to uniezależnia historię od restartów backendu.

B. BACKEND: ASP.NET Core (Web API - .NET 8.0)
   - Odgrywa rolę mostu pośredniczącego (Proxy/Orkiestratora) pomiędzy systemem
     operacyjnym Windows, lokalnym serwerem Ollama a frontendem React.
   - Wykorzystuje model "Minimal APIs", serwując punkty końcowe przy użyciu
     lekkiego, wbudowanego serwera Kestrel na porcie 8000.
   - Usługi rejestrowane w kontenerze Dependency Injection (DI) jako Singletony:
     * SystemInfoService: Pobiera statyczne informacje o sprzęcie (rejestr
        Windows, zapytania WMI przez System.Management) oraz dynamiczne parametry
        użycia procesora i RAM (obciążenie globalne i na rdzeń). Udostępnia również
        metodę GetNvidiaSmiAsync() do próbkowania metryk GPU przez nvidia-smi
        oraz GetSystemSnapshot() do szybkiej migawki RAM/CPU podczas benchmarku.
      * OllamaClientService: Klient HTTP zarządzający odpytywaniem lokalnego API
        Ollamy, przetwarzaniem odpowiedzi oraz generowaniem raportów eksperckich.
        Metoda RunLlmBenchmarkAsync() próbkuje metryki GPU i systemu przed/po
        teście wnioskowania i zwraca rozszerzony obiekt OllamaResult.

C. LOKALNY SILNIK LLM: Ollama (Local AI Engine)
   - Lokalny serwer uruchamiający modele LLM (np. Llama 3.1, Phi-3, Mistral),
     działający w tle na porcie 11434.
   - Narzędzie to odpowiada za wnioskowanie na lokalnym sprzęcie (wykorzystując
     akcelerację GPU poprzez CUDA/ROCm lub instrukcje procesora CPU).

D. Pozyskiwanie danych telemetrycznych przez WMI (Windows Management Instrumentation)
   Backend odpytuje niskopoziomowe interefesy systemowe przy użyciu biblioteki
   System.Management. Pozwala to na pobranie parametrów trudnodostępnych z poziomu
   samej platformy .NET, takich jak:
   - Klasa "Win32_VideoController": Służy do pobrania nazwy dedykowanej karty
     graficznej (GPU) oraz jej pamięci podręcznej (VRAM).
   - Klasa "Win32_Processor": Służy do pobrania liczby rdzeni fizycznych procesora.
   - Bezpośredni odczyt z rejestru systemowego Windows: Dokładny model i nominalna
     częstotliwość taktowania procesora pobierana jest ze ścieżki rejestru:
     HKEY_LOCAL_MACHINE\HARDWARE\DESCRIPTION\System\CentralProcessor\0.

E. Funkcjonalności eksportu danych i raportów (HistoryPanel):
   Frontend w panelu historii udostępnia pełne zarządzanie zapisanymi danymi bez potrzeby
   ich przetwarzania przez backend:
   - Eksport do formatu Markdown (*.md): Generuje plik tekstowy na podstawie właściwości
     `ai_report` w obiekcie sesji, umożliwiając łatwy odczyt raportu w edytorach tekstu.
   - Eksport do formatu JSON (*.json): Umożliwia pobranie pełnego obiektu `BenchmarkRun`
     (ze szczegółową specyfikacją techniczną komputera i wynikami testu) do celów
     diagnostycznych lub migracji danych.
   - Trwałe oczyszczanie: Użytkownik może usunąć wybrane rekordy z `localStorage` za pomocą
     filtrowania po unikalnym identyfikatorze GUID generowanym po stronie backendu.

2. INTERFEJSY API (KOMUNIKACJA FRONTEND <-> BACKEND)
-----------------------------------------------------------------------------

Serwer ASP.NET Core wystawia następujące endpointy HTTP/SSE dla frontendu:

1. GET /api/system-info
   - Zwraca szczegółową statyczną specyfikację maszyny.
   - Zwracany obiekt (JSON): SystemSpecs (OS, model CPU, rdzenie, RAM, detale GPU).

2. GET /api/system-info/live
   - Cykliczny punkt odczytu obciążenia sprzętu (odpytywany przez frontend co 1.5s).
   - Zwracany obiekt (JSON): LiveMetrics (użycie CPU w %, lista obciążenia rdzeni,
     użycie RAM w GB, procent RAM).

3. GET /api/models
   - Pobiera listę modeli zainstalowanych lokalnie w Ollamie.
   - Pobiera dane bezpośrednio z Ollamy i przesyła je w ustrukturyzowanej formie.

4. POST /api/benchmark/run-stream
   - Uruchamia test wydajności w formie Server-Sent Events (SSE).
   - Body żądania (RunBenchmarkRequest): { model, tests: ["ollama"], complexity }
   - Format wysyłanych zdarzeń SSE (każdy event):
     data: {"status":"running","step":"ollama","message":"...","progress":20}
     (dwie nowe linie \n\n kończące blok zdarzenia)
   - Ostatni event: status="completed", step="finished", progress=100,
     pole "run" zawiera kompletny obiekt BenchmarkRun.
   - Nagłówki odpowiedzi: Content-Type: text/event-stream, Cache-Control: no-cache,
     Connection: keep-alive, X-Accel-Buffering: no (wyłącza buforowanie proxy).
   - UWAGA: Używa bezpośredniego context.Response.WriteAsync() zamiast StreamWriter,
     aby uniknąć błędu ERR_INCOMPLETE_CHUNKED_ENCODING (StreamWriter z blokiem using
     dispose'uje strumień HTTP zbyt wcześnie).

5. Zabezpieczenia i protokoły komunikacyjne:
   - Zabezpieczenie CORS (Cross-Origin Resource Sharing):
     Ponieważ frontend (port 5173) działa na innym porcie niż backend (port 8000),
     serwer ASP.NET Core konfiguruje politykę CORS zezwalającą na żądania z pochodzenia
     http://localhost:5173 przy użyciu dowolnych metod (GET/POST) i nagłówków. Zapobiega
     to blokowaniu zapytań przez przeglądarki.
   - Server-Sent Events (SSE) vs WebSockets:
     Strumieniowanie logów z benchmarku LLM zostało zrealizowane przez jednokierunkowe
     SSE (MediaType: text/event-stream). Protokół ten jest prostszy i bardziej niezawodny
     od WebSockets dla scenariuszy, w których dane płyną wyłącznie od serwera do klienta,
     posiadając przy tym wbudowaną w przeglądarkę automatyczną obsługę reconnectu.
   - Bezpieczeństwo wątkowe w telemetrii:
     Dla odczytów obciążenia CPU zaimplementowano buforowanie stanu w serwisie Singleton,
     tak aby setki cyklicznych zapytań z przeglądarki nie przeciążały systemu operacyjnego
     nieustannym, kosztownym odpytywaniem liczników wydajności Windows w tym samym momencie.

3. HTTP API OLLAMA - STRUKTURY ZAPYTAŃ I SPOSÓB KOMUNIKACJI
-----------------------------------------------------------------------------

Serwis OllamaClientService komunikuje się z Ollamą za pomocą natywnego API HTTP.

A. Pobieranie listy modeli
   - Metoda: GET
   - Endpoint: http://localhost:11434/api/tags
   - Odpowiedź: JSON zwierający tablicę obiektów opisujących modele (nazwa, rozmiar).

B. Uruchomienie wnioskowania (Test benchmarkowy)
   - Metoda: POST
   - Endpoint: http://localhost:11434/api/generate
   - Request Body (JSON):
     {
       "model": "llama3.1:latest",
       "prompt": "[Treść promptu zależna od poziomu trudności]",
       "stream": false,
       "options": {
         "num_predict": 120,
         "temperature": 0.0
       }
     }
      * stream: Ustawione na false, aby otrzymać jedną kompletną odpowiedź wraz
        z pełnymi metrykami czasu wykonania w nanosekundach.
      * num_predict: Limit generowania tokenów. W zależności od profilu trudności
        wynosi odpowiednio: 1200 (szybki), 6000 (średni), 15000 (złożony).
      * temperature: 0.1 / 0.4 / 0.7 w zależności od złożoności (wyższa temp.
        dla dłuższych promptsów, niższa zapewnia bardziej deterministyczne wyniki).

C. Struktura odpowiedzi z Ollama (/api/generate)
   Po zakończeniu generowania, Ollama zwraca szczegółowe metryki wydajności:
   {
     "model": "llama3.1:latest",
     "response": "Wygenerowany tekst...",
     "total_duration": 4820100200,      // Całkowity czas w nanosekundach
     "load_duration": 1200300,          // Czas ładowania modelu w ns
     "prompt_eval_count": 18,           // Liczba tokenów w prompcie
     "prompt_eval_duration": 250100300, // Czas przetwarzania promptu w ns
     "eval_count": 89,                  // Liczba wygenerowanych tokenów
     "eval_duration": 4560000100        // Czas generowania tokenów w ns
   }

D. Mechanizm generowania raportów eksperckich (AI Analyst)
   Po zakończeniu testu wydajności, backend automatycznie inicjuje drugie, dedykowane
   zapytanie (POST /api/generate) do lokalnej instancji Ollama, aby wygenerować
   ekspercki raport techniczny na podstawie zgromadzonych metryk.
   1. Dynamiczna synteza promptu (Wstrzykiwanie telemetrii):
      Do modelu wysyłany jest sprofilowany prompt w języku polskim. Backend dynamicznie
      wstrzykuje do niego pełne statyczne parametry komputera (system operacyjny, model CPU,
      fizyczna i logiczna liczba rdzeni, pamięć RAM, nazwy kart graficznych) oraz uzyskane
      chwilę wcześniej wyniki benchmarku (t/s, TTFT, nazwa testowanego modelu).
   2. Dostosowanie profilu raportu do poziomu trudności (złożoności):
      Instrukcje systemowe i parametry generowania są modyfikowane na backendzie:
      - SZYBKI (Quick): Żądanie maksymalnie 3-zdaniowej, zwięzłej oceny.
        * Ograniczenie tokenów (num_predict): 150. Temperatura: 0.1.
      - ŚREDNI (Medium): Szczegółowa ocena (2-3 akapity) ze wskazaniem wąskich gardeł
        i podstawowych rekomendacji dotyczących rozbudowy sprzętu.
        * Ograniczenie tokenow (num_predict): 500. Temperatura: 0.3.
      - ZAAWANSOWANY (Complex): Pełny, wielosekcyjny raport ekspercki w formacie Markdown,
        zawierający tabele alokacji VRAM/RAM, analizę balansu obciążenia CPU/GPU
        oraz dedykowany plan uaktualnienia sprzętu.
        * Ograniczenie tokenow (num_predict): 1200. Temperatura: 0.5.

E. Optymalizacja i akceleracja GPU w Ollama
   W zapytaniach do API Ollamy program celowo przekazuje parametry optymalizacyjne:
   - "num_gpu": -1 — nakazuje silnikowi Ollama automatyczną detekcję dostępnych układów GPU
     (np. CUDA, ROCm) i załadowanie jak największej liczby warstw modelu do szybkiej pamięci VRAM.
   - "temperature": 0.0 / 0.1 — wyciszenie kreatywności modelu podczas właściwego benchmarku.
     Gwarantuje to powtarzalność pomiarów (LLM wybiera zawsze najbardziej prawdopodobny token,
     co redukuje różnice czasu wnioskowania w kolejnych próbach).

4. MATEMATYCZNE OBLICZANIE METRYK WYDAJNOŚCI LLM
-----------------------------------------------------------------------------

Na podstawie danych otrzymanych z Ollamy, backend ASP.NET Core oblicza metryki
wydajności według następujących wzorów:

1. Całkowity czas generowania (Total Generation Time [s]):
   TotalTimeSec = total_duration / 1,000,000,000.0

2. Opóźnienie do pierwszego tokenu — TTFT (Latency [s]):
   LatencySec = (load_duration + prompt_eval_duration) / 1,000,000,000.0
   (Czas ładowania modelu + czas przetworzenia promptu = czas do pierwszego tokenu)

3. Prędkość generowania tekstu (Tokens per Second [t/s]):
   TokensPerSec = eval_count / (eval_duration / 1,000,000,000.0)

4. Prędkość przetwarzania promptu wejściowego (Prompt Eval Tokens per Second [t/s]):
   PromptEvalTokensPerSec = prompt_eval_count / (prompt_eval_duration / 1 000 000 000.0)

5. Czas ładowania modelu do pamięci (Load Duration [s]):
   LoadDurationSec = load_duration / 1 000 000 000.0

6. Czas przetwarzania promptu (Prompt Eval Duration [s]):
   PromptEvalDurationSec = prompt_eval_duration / 1 000 000 000.0

7. Czas generowania odpowiedzi (Eval Duration [s]):
   EvalDurationSec = eval_duration / 1 000 000 000.0

Wszystkie pola są mapowane na rozszerzony model OllamaResult i przesyłane do
frontendu jako część obiektu BenchmarkRun (pole results.ollama).

4b. METRYKI GPU I SYSTEMU (nvidia-smi + WMI) — POBIERANE PODCZAS BENCHMARKU
-----------------------------------------------------------------------------

Oprócz metryk Ollamy, backend próbkuje również stan sprzętu w momencie testu.

A. Metryki GPU — nvidia-smi (SystemInfoService.GetNvidiaSmiAsync)
   Metoda wywołuje zewnętrzne narzędzie nvidia-smi z parametrami:
   > nvidia-smi --query-gpu=memory.used,memory.total,utilization.gpu,
               power.draw,power.limit,temperature.gpu
               --format=csv,noheader,nounits

   Zwracane wartości (pobierane PO zakończeniu wnioskowania):
   - gpu_vram_used_mb     : Użyta pamięć VRAM podczas testu [MB]
   - gpu_vram_total_mb    : Całkowita pamięć VRAM karty [MB]
   - gpu_util_percent     : Obciążenie rdzeni GPU [%]
   - gpu_power_draw_w     : Chwilowy pobór mocy karty [W]
   - gpu_power_limit_w    : Limit mocy TDP karty [W]
   - gpu_temp_c           : Temperatura rdzenia GPU [°C]
   - gpu_metrics_available: true/false — czy nvidia-smi było dostępne na tym systemie

   Uwaga: Jeśli użytkownik nie ma karty NVIDIA lub nie ma zainstalowanych sterowników
   NVIDIA, pole gpu_metrics_available = false i sekcja GPU w UI wyświetla komunikat
   o niedostępności.

B. Metryki systemowe — WMI (SystemInfoService.GetSystemSnapshot)
   Metoda wywołuje krótkie zapytania WMI po zakończeniu testu:
   - Win32_OperatingSystem: Pobiera użytą i całkowitą pamięć RAM.
   - Win32_PerfFormattedData_PerfOS_Processor WHERE Name=\'_Total\': Pobiera
     łączne obciążenie CPU.

   Zwracane wartości:
   - sys_ram_used_gb  : Pamięć RAM używana podczas testu [GB]
   - sys_ram_total_gb : Całkowita pamięć RAM systemu [GB]
   - sys_ram_percent  : Procentowe zajęcie RAM [%]
   - sys_cpu_percent  : Łączne obciążenie procesora podczas testu [%]

4c. PREDYKCJA WYDAJNOŚCI (TPS) PRZED BENCHMARKIEM (CompatibilityPanel.jsx)
-----------------------------------------------------------------------------

Aplikacja posiada interaktywny kalkulator zgodności (CompatibilityPanel.jsx), który prognozuje wydajność wnioskowania (tokeny/s) dla wybranego modelu zanim użytkownik fizycznie uruchomi benchmark. Zamiast sztywnych baz danych z nazwami modeli GPU (co pomijałoby nowe lub niestandardowe karty), zastosowano fizyczno-sprzętowy model matematyczny:

A. Fizyczne założenia limitacji przepustowości (Memory-Bandwidth Bound):
   - Wnioskowanie na modelach LLM polega na sekwencyjnym czytaniu całego stanu sieci z pamięci VRAM do rejestrów GPU przy każdym wygenerowanym tokenie.
   - Prawdziwym fizycznym ograniczeniem wydajności jest przepustowość pamięci karty graficznej (Memory Bandwidth [GB/s]), a nie czysta moc obliczeniowa rdzeni (TFLOPS).
   - Istnieje silna, naturalna korelacja między rozmiarem pamięci VRAM a szerokością szyny i przepustowością karty (szyna 128-bit dla kart 4GB, 192-bit dla 12GB, 384-bit dla 24GB). Średni współczynnik to:
     Przepustowość (GB/s) ≈ 40 * Rozmiar VRAM (GB)
   - Rzeczywista sprawność silnika wnioskującego (llama.cpp) wynosi średnio około 33-36% fizycznej przepustowości szyny pamięci z uwagi na opóźnienia kerneli obliczeniowych i narzut systemu.

B. Matematyczny wzór estymacyjny:
   1. Klasa wydajności GPU (gpuPower):
      gpuPower = Math.round(activeGpu.vram_gb * 16.5) (dla kart dedykowanych z VRAM >= 2.0 GB)
      gpuPower = 35 (wartość bazowa dla zintegrowanych układów graficznych)
   2. Prognozowana prędkość generowania (estTps):
      estTps = gpuPower / (ModelSizeB * 1.1)
      Co po podstawieniu daje uproszczoną postać fizyczną:
      estTps = (15 * VRAM [GB]) / RozmiarModelu [Miliardy parametrów]

   Ten wzór zapewnia bardzo wysoką zgodność z realnymi testami na różnym sprzęcie:
   - GTX 1650 (4 GB VRAM) dla modelu 3B: prognoza ~20.0 t/s (realnie ~16-22 t/s)
   - RTX 3060 (12 GB VRAM) dla modelu 3B: prognoza ~60.0 t/s (realnie ~45-65 t/s)
   - RTX 4090 (24 GB VRAM) dla modelu 3B: prognoza ~120.0 t/s (realnie ~100-140 t/s)

C. Wyświetlanie w UI (BenchmarkPanel.jsx)
   Po zakończeniu testu, wszystkie metryki są prezentowane w trójsegmentowej
   tabeli (odpowiednik /set verbose w Ollama CLI):
   - Sekcja 1 (niebieski nagłówek): Czasy wnioskowania Ollamy (verbose)
   - Sekcja 2 (fioletowy nagłówek): GPU z nvidia-smi (lub info o braku NVIDIA)
   - Sekcja 3 (zielony nagłówek): Systemowe RAM i CPU (migawka WMI)

   Każdy wiersz tabeli (komponent MetricRow) zawiera teraz:
   1. Krótki opis (np. "Całkowity czas od wysłania pytania do zakończenia odpowiedzi").
   2. Interpretację wyniku - wskazówkę ułatwiającą ocenę wydajności sprzętu:
      - "Im więcej, tym lepiej ⬆️" (np. dla prompt eval rate, eval rate, obciążenia GPU).
      - "Im mniej, tym lepiej ⬇️" (np. dla total duration, load duration, prompt eval
        duration, eval duration, temperatury GPU, zużycia RAM, obciążenia CPU).
      - "Zależy od konfiguracji ➔" (np. dla liczby tokenów promptu/odpowiedzi,
        zużycia VRAM, poboru mocy).

5. KOMENDY UŻYWANE W PROJEKCIE (ŚCIĄGAWKA)
-----------------------------------------------------------------------------

A. FRONTEND (katalog client)
   1. Instalacja bibliotek i zależności z package.json:
      > npm install
   2. Uruchomienie lokalnego serwera deweloperskiego Vite:
      > npm run dev
      (Domyślnie uruchamia się pod adresem: http://localhost:5173)
   3. Skompilowanie zoptymalizowanej wersji produkcyjnej HTML/JS/CSS:
      > npm run build
   4. Uruchomienie statycznego podglądu zbudowanej wersji produkcyjnej:
      > npm run preview

B. BACKEND (katalog server)
   1. Kompilacja projektu i sprawdzenie błędów składni C#:
      > dotnet build
   2. Uruchomienie serwera ASP.NET Core:
      > dotnet run
      (Backend uruchamia się i nasłuchuje pod adresem: http://127.0.0.1:8000)

C. SERWER OLLAMA (System operacyjny)
   1. Sprawdzenie statusu i listy pobranych modeli lokalnych w terminalu:
      > ollama list
   2. Pobranie nowego modelu LLM z rejestru Ollama (np. Llama 3.1):
      > ollama pull llama3.1
   3. Ręczne uruchomienie konwersacji z modelem bezpośrednio w konsoli:
      > ollama run llama3.1
   4. Sprawdzenie uruchomionych w danej chwili modeli w pamięci RAM/VRAM:
      > ollama ps

6. CHARAKTERYSTYKA SYSTEMU, WYMAGANIA I SPOSÓB URUCHOMIENIA
-----------------------------------------------------------------------------

A. Architektura Klient-Serwer:
   Aplikacja została zaprojektowana w nowoczesnej architekturze rozproszonej:
   - Klient (Frontend - React): Uruchamia się w przeglądarce internetowej użytkownika.
     Pobiera statyczne pliki z serwera Vite i komunikuje się asynchronicznie (AJAX,
     Server-Sent Events) z serwerem API. Nie posiada własnej bazy danych po stronie
     serwera - przechowuje historię lokalnie w pamięci przeglądarki (localStorage).
   - Serwer (Backend - ASP.NET Core): Działa jako dedykowana usługa systemowa
     udostępniająca interfejs API na porcie 8000. Komunikuje się z niskopoziomowymi
     bibliotekami systemu Windows (WMI) w celu odczytu parametrów podzespołów
     sprzętowych oraz przekazuje zapytania do lokalnej instancji Ollama.
   - Serwer LLM (Ollama): Działa w tle jako niezależna usługa systemowa,
     obsługując żądania wnioskowania modeli sztucznej inteligencji.

B. Wymagania Sprzętowe i Programowe:
   - Wymagania sprzętowe (Rekomendowane):
     * Procesor: Intel Core i5/i7 lub AMD Ryzen 5/7 (min. 6 rdzeni fizycznych).
     * Pamięć RAM: Minimum 16 GB (zalecane 32 GB dla stabilnego działania modeli 8B).
     * Karta Graficzna (GPU): Karta dedykowana NVIDIA GeForce (seria GTX 10- lub nowsza)
       z minimum 6 GB pamięci VRAM (np. RTX 3060/4060) w celu uzyskania satysfakcjonującej
       wydajności (prędkości generowania tokenów powyżej 15 t/s).
     * Dysk twardy: SSD z co najmniej 15-30 GB wolnego miejsca na lokalne modele LLM.
    - Wymagania programowe (Systemowe):
      * System operacyjny: Microsoft Windows 10 lub 11 (64-bit).
      * Środowisko .NET SDK 8.0 (lub nowsze) zainstalowane lokalnie.
      * Środowisko Node.js (wersja LTS, zalecana 18.x lub nowsza).
      * Zainstalowany i uruchomiony serwer Ollama w wersji dla systemu Windows.
      * nvidia-smi — zewnętrzne narzędzie systemowe NVIDIA (nie jest
        to biblioteka NuGet ani pakiet npm). Jest instalowane automatycznie razem
        ze sterownikami karty graficznej NVIDIA (CUDA Driver). Backend wywołuje je
        przez System.Diagnostics.Process.Start() z parametrem --query-gpu w celu
        pobrania metryk GPU (VRAM, obciążenie, temperatura, pobór mocy) podczas
        testu wydajności. Analogia: tak jak Ollama jest zewnętrzną usługą systemową
        uruchamianą jako proces, tak nvidia-smi jest zewnętrznym narzędziem CLI
        wywoływanym jednorazowo z poziomu kodu backendu.
        Jeśli karta nie jest NVIDIA lub sterowniki NVIDIA nie są zainstalowane,
        pole gpu_metrics_available = false i sekcja GPU jest pomijana w UI.

C. Sposób Instalacji:
   1. Sklonuj lub rozpakuj archiwum z kodem źródłowym do wybranego katalogu (np. `D:\Projekt LLM DOTNET`).
   2. Pobierz i zainstaluj oficjalny instalator Ollama ze strony: https://ollama.com.
   3. Pobierz i zainstaluj środowisko .NET 8.0 SDK ze strony Microsoft.
   4. Pobierz i zainstaluj środowisko Node.js ze strony nodejs.org.
   5. Otwórz terminal w katalogu `client` i zainstaluj zależności frontendu komendą:
      > npm install

D. Sposób Aktywacji (Uruchomienia):
   1. Upewnij się, że usługa Ollama działa w tle. Pobierz przykładowy lekki model (np. Llama 3.1 lub Phi-3) komendą w terminalu systemowym:
      > ollama pull llama3.1
   2. Aktywacja backendu (.NET):
      Otwórz terminal w katalogu `server/` i wpisz:
      > dotnet run
      Backend skompiluje się i uruchomi na porcie 8000.
   3. Aktywacja frontendu (React):
      Otwórz osobne okno terminala w katalogu `client/` i wpisz:
      > npm run dev
      Aplikacja deweloperska uruchomi się pod adresem http://localhost:5173.
   4. Otwórz przeglądarkę internetową i wejdź na podany adres: http://localhost:5173.

7. PROCES WERYFIKACJI I WALIDACJI (V&V)
-----------------------------------------------------------------------------

W celu zapewnienia wysokiej jakości oprogramowania oraz zgodności z założeniami
funkcjonalnymi, w projekcie wdrożono dwutorowy proces kontroli jakości:

A. Weryfikacja (Verification) – "Czy tworzymy produkt poprawnie?"
   Skupia się na technicznej poprawności kodu, standardach programistycznych
   i spójności struktur danych:
   - Statyczna Analiza i Kompilacja (C# / .NET):
     * Wykorzystanie kompilatora Roslyn (`dotnet build`) do weryfikacji poprawności
       typów, składni języka C# oraz powiązań referencyjnych między klasami.
     * Podział modeli danych na osobne pliki został zweryfikowany pod kątem braku
       konfliktów przestrzeni nazw (`ProjektAI.Backend.Models`).
   - Weryfikacja Standardów Kodu (JavaScript / React):
     * Wykorzystanie narzędzia ESLint (`npm run lint`) w celu detekcji błędów
       w czasie rzeczywistym, weryfikacji nieużywanych zmiennych oraz egzekwowania
       szczególnych zasad React Hooks (np. prawidłowych tablic zależności w useEffect).
   - Kontrakt Strukturalny DTO:
     * Automatyczna walidacja formatu wymiany danych za pomocą natywnej biblioteki
       `System.Text.Json`. Jeśli dane nadesłane z Ollamy lub z frontendu nie są
       zgodne z atrybutami `[JsonPropertyName]`, proces zgłasza wyjątek na poziomie
       deserializacji, co uniemożliwia przetwarzanie uszkodzonych pakietów.
   - Obsługa błędów (Error Handling):
     * Weryfikacja stanów wyjątkowych w kodzie backendu (np. bloki `try-catch` przy
       komunikacji HTTP z Ollama w `OllamaClientService.cs`), zapobiegająca awarii
       całej aplikacji w przypadku utraty połączenia.

B. Walidacja (Validation) – "Czy tworzymy właściwy produkt?"
   Skupia się na weryfikacji użyteczności aplikacji oraz poprawności merytorycznej
   realizowanych przez nią zadań benchmarkowych i kalkulacyjnych:
   - Ciągła Walidacja Połączeń (Live Checker):
     * Pętla pingu we frontendzie cyklicznie (co 5s) weryfikuje łączność z serwerem
       ASP.NET Core, a ten z kolei sprawdza status usługi Ollama (`/api/tags`).
       Użytkownik otrzymuje natychmiastową wizualną informację w pasku statusu
       o sprawności całego ekosystemu.
   - Walidacja Wyników Pomiary Wydajności LLM:
     * Wyłączenie strumieniowania (`"stream": false`) w zapytaniach do Ollamy gwarantuje,
       że czasy generowania tokenów są pobierane bezpośrednio z wewnętrznych liczników
       silnika Ollama (w nanosekundach). Eliminuje to błędy pomiarowe (narzut sieciowy,
       opóźnienia renderowania DOM na kliencie), dając fizyczny, obiektywny wynik.
   - Walidacja Algorytmu Kompatybilności Sprzętowej:
     * Kalkulator w `CompatibilityPanel.jsx` dynamicznie wyznacza zapotrzebowanie
       modelu na pamięć VRAM na podstawie wielkości parametrów (np. 8B) oraz
       wybranej kwantyzacji (np. Q4_K_M zajmujący ok. 4.8 GB + narzut kontekstu 0.8 GB).
       Wzór ten jest zestawiany z rzeczywistą ilością pamięci VRAM pobraną
       z fizycznej karty graficznej użytkownika za pośrednictwem zapytań WMI,
       co pozwala na trafną walidację decyzji o offloadingu modelu.
   - Testy UX i Interakcji Użytkownika:
     * Walidacja zachowania interfejsu w sytuacjach krytycznych — np. przycisk
       uruchomienia testu wydajności jest blokowany w stanie aktywnym, aby zapobiec
       wysłaniu podwójnego żądania i zafałszowaniu wyników obciążenia procesora.
     * Dynamiczne skalowanie osi czasu wykresów SVG w Dashboardzie zapewnia czytelną
       prezentację danych telemetrycznych niezależnie od obciążenia.

8. PODSUMOWANIE I WNIOSKI
-----------------------------------------------------------------------------

Projekt Projekt AI stanowi w pełni funkcjonalne, wydajne i nowoczesne narzędzie do lokalnego benchmarkowania modeli językowych LLM oraz monitorowania zasobów sprzętowych komputera w czasie rzeczywistym.

Główne wnioski z realizacji i działania systemu:
1. Skuteczność lokalnego uruchamiania AI: Użycie platformy Ollama pozwala na w pełni autonomiczne i darmowe uruchamianie modeli LLM klasy konsumenckiej (takich jak Phi-3, Llama 3.1 8B) bezpośrednio na komputerze użytkownika, bez konieczności przesyłania danych do chmurowych API (zapewniając 100% prywatności).
2. Doskonała integracja .NET i React: Połączenie szybkiego backendu ASP.NET Core (który ma bezpośredni, niskopoziomowy dostęp do zasobów Windows poprzez WMI) z reaktywnym interfejsem React zapewnia użytkownikowi płynne i profesjonalne doświadczenie (UX) z dynamicznymi wykresami SVG oraz natychmiastowym czasem reakcji.
3. Elastyczność przechowywania danych (localStorage): Rezygnacja z bazy danych po stronie serwera na rzecz przechowywania historii bezpośrednio w przeglądarce (`localStorage`) uprościła architekturę backendu, eliminując problem blokowania plików oraz ułatwiając migrację i przenoszenie aplikacji.
4. Krytyczne znaczenie pamięci VRAM: Przeprowadzona walidacja kalkulatora kompatybilności jednoznacznie wskazuje, że wydajność modeli językowych jest bezpośrednio uzależniona od obecności dedykowowanego układu graficznego (GPU) i odpowiedniej ilości VRAM. Uruchomienie modelu w całości w pamięci karty graficznej (GPU) skutkuje kilkukrotnie wyższą prędkością generowania tokenów w porównaniu do trybu mieszanego lub samego procesora (CPU).
5. Potencjał rozwojowy: Modułowa architektura z podziałem na dedykowane klasy modeli DTO oraz wstrzykiwane usługi Singleton ułatwia ewentualną przyszłą rozbudowę projektu o dodatkowe silniki testowe (np. benchmarki pamięci dyskowej, testy kart graficznych innych producentów czy wsparcie dla API chmurowych do celów porównawczych).
